# API Versioning Strategy

> **Document type:** Architecture Decision Record
> **ADR:** ADR-019
> **Version:** 1.0.0
> **Last updated:** 2026-03-19
> **Scope:** All phases (P1–P5)

---

## Decision

Use **URL-based versioning** with a single `/api/v1` prefix for all P1–P5 endpoints. A new version (`/api/v2`) is introduced only when existing response shapes have breaking changes.

---

## Context

The Healthcare Clinic platform is developed incrementally across 5 phases. Each phase adds new endpoints (e.g., P2 adds `/shifts`, P3 adds `/video-sessions`). The question is how to version these additions and handle breaking changes over time.

**API versioning approaches considered:**

| Approach | Example | Pros | Cons |
|----------|---------|------|------|
| URL versioning ✓ | `/api/v1/bookings` | Simple, explicit, cacheable | URL changes on major version |
| Header versioning | `Accept: application/vnd.clinic.v1+json` | Clean URLs | Hard to test in browser, proxy complexity |
| Query parameter | `/bookings?version=1` | Easy to add | Non-standard, caching issues |

---

## Rules

### 1. All P1–P5 endpoints live under `/api/v1`

New features are **additive** — they do not break existing consumers. Adding `/api/v1/shifts` (P2) or `/api/v1/video-sessions` (P3) does not require a version bump.

### 2. Non-breaking changes stay in v1

| Change type | Breaking? | Version bump? |
|------------|-----------|--------------|
| Add a new endpoint | No | No |
| Add an optional field to a response | No | No |
| Add an optional query parameter | No | No |
| Add a new error code | No | No |
| Remove a field from a response | **Yes** | **Yes → v2** |
| Rename a field in a response | **Yes** | **Yes → v2** |
| Change a field's type | **Yes** | **Yes → v2** |
| Remove an endpoint | **Yes** | **Yes → v2** |

### 3. When v2 is introduced

If a breaking change is absolutely necessary (e.g., restructuring the response envelope for P5 multi-tenancy), introduce `/api/v2` with:

- **v1 and v2 coexist** for a deprecation period
- **Sunset header** on v1 responses: `Sunset: Sat, 01 Jan 2028 00:00:00 GMT`
- **Deprecation period:** minimum 6 months
- **v1 removal:** only after all known clients have migrated

### 4. OpenAPI spec per version

Each version has its own OpenAPI spec:
- `openapi-v1.json` — generated by NestJS Swagger
- `openapi-v2.json` — generated when v2 exists

Frontend apps pin to a specific version and regenerate their `@clinic/api-client` from the corresponding spec.

---

## Consequences

- **Simple:** One URL prefix to remember and test for the first 5 phases
- **Low overhead:** No header parsing, no media type negotiation
- **Forward compatible:** Adding v2 requires setting up a second versioned controller module in NestJS, which is well-supported via `@Controller({ version: '2' })`
- **Client impact:** If v2 is introduced, all 3 frontend apps (dashboard, member, staff) must be updated to use v2 endpoints. The shared `@clinic/api-client` package makes this a single-package update.

---

## NestJS Versioning Setup (when v2 is needed)

```typescript
// main.ts
app.enableVersioning({
  type: VersioningType.URI,
  prefix: 'api/v',
  defaultVersion: '1',
});

// Controller with explicit version
@Controller({ path: 'bookings', version: '2' })
export class BookingsControllerV2 {
  // New response shapes for v2
}
```
